# 代码审查要求

## 审查清单

### 功能性

- [ ] 功能实现符合需求
- [ ] 边界条件处理正确
- [ ] 错误处理完善
- [ ] 无明显的逻辑错误

### 代码质量

- [ ] 遵循项目代码规范
- [ ] 命名清晰有意义
- [ ] 无重复代码（DRY 原则）
- [ ] 适当的注释（中文）
- [ ] 函数/方法长度适中
- [ ] 复杂度可控

### 安全性

- [ ] 无 SQL 注入风险
- [ ] 输入验证完善
- [ ] 敏感数据正确处理（不记录明文密码、API Key）
- [ ] 权限控制正确
- [ ] 无 XSS 漏洞

### 性能

- [ ] 无 N+1 查询问题
- [ ] 合理使用缓存
- [ ] 避免内存泄漏
- [ ] 大数据量处理合理

### 测试

- [ ] 新增功能有对应测试
- [ ] 测试覆盖率符合要求
- [ ] 所有测试通过
- [ ] 边界情况有测试

---

## 覆盖率要求

| 测试类型 | 后端 | 前端 |
|---------|------|------|
| 单元测试 | > 80% | > 70% |
| 集成测试 | 核心流程 | 核心流程 |
| E2E 测试 | - | 关键路径 |

## PR 模板要求

创建 PR 时必须包含以下内容：

```markdown
## 变更说明
<!-- 描述本次 PR 的主要变更 -->

## 关联 Issue
<!-- 关联的 Issue 编号，如：Closes #123 -->

## 测试说明
<!-- 如何测试这些变更 -->
- [ ] 单元测试已添加/更新
- [ ] 集成测试通过
- [ ] 手动测试通过

## 截图
<!-- 如有 UI 变更，添加前后对比截图 -->

## 检查清单
- [ ] 代码通过 lint 检查
- [ ] 测试全部通过
- [ ] 文档已更新（如有必要）
```

## 审查原则

### 对审查者

1. **及时响应**: 尽快完成审查，避免阻塞
2. **建设性反馈**: 提出问题同时给出建议
3. **区分优先级**: 明确区分阻塞性问题和建议
4. **尊重作者**: 保持专业和礼貌

### 对作者

1. **小而专注**: 保持 PR 规模适中，便于审查
2. **自测充分**: 提交前充分自测
3. **清晰描述**: 提供清晰的 PR 描述
4. **积极回应**: 及时响应审查意见

---

## 阻塞性问题 vs 建议

### 阻塞性问题（必须修复）

- 功能不符合需求
- 安全漏洞
- 明显的性能问题
- 测试缺失或失败
- 破坏向后兼容性

### 建议（可选修复）

- 代码风格偏好
- 可选的重构建议
- 文档改进建议
- 命名建议

## 常见审查问题

### Go 后端

| 问题 | 建议 |
|------|------|
| 错误未包装 | 使用 `fmt.Errorf("%w", err)` 添加上下文 |
| 缺少注释 | 导出函数必须添加中文注释 |
| 硬编码值 | 提取为常量或配置 |
| 过长函数 | 拆分为多个小函数 |
| 未处理错误 | 始终检查并处理错误 |

### TypeScript 前端

| 问题 | 建议 |
|------|------|
| 类型定义缺失 | 为所有 props 和返回值定义类型 |
| any 类型 | 避免使用 any，使用具体类型 |
| 魔法字符串 | 提取为常量或枚举 |
| 重复样式 | 提取为 CSS 类或样式变量 |
| 未使用的导入 | 删除未使用的导入 |
